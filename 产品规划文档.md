## 执行摘要

本文档规划了一条由开源社区驱动、以非营利原则运作的资产管理产品线，涵盖从基础设施层到应用层的 8 大核心模块：

- **基础设施层**: KMS（密钥管理服务）、HMS（硬件管理服务）
- **核心技术层**: MPC（多方安全计算）
- **钱包产品线**: 自托管钱包、多签钱包、智能钱包、托管钱包
- **硬件产品线**: 硬件钱包

### 核心战略

1. **技术路线**: 以开源 MPC 技术为核心差异化竞争力
2. **社区策略**: 先夯实基础设施，再逐步扩展到钱包与硬件场景，由社区需求驱动优先级
3. **产品策略**: 模块化设计，保证自托管、可 fork、可复用
4. **治理策略**: 安全与合规前置，采用公开提案 + 多签治理模式

### 品牌介绍

**KashGuard = Key + Cash + Guard**  
寓意以密码学守护用户的“钥匙”和“资产”，通过可信 Guard 体系构建端到端安全的资产管理体验。

## 产品线详细规划

### 阶段一：基础设施层（6-12个月）

#### 1. KMS（密钥管理服务）

**产品定位**
- 核心基础设施，为所有其他产品提供密钥管理能力
- 企业级密钥全生命周期管理平台

**目标客户**
- 金融机构（银行、保险、证券）
- 云服务提供商
- 大型企业（需要密钥管理合规）

**核心功能**
- 密钥生成、存储、分发、轮换、销毁
- 对称密钥和非对称密钥支持
- HSM 集成（PKCS#11）
- 密钥策略和访问控制
- 审计日志和合规报告
- 高可用架构（99.99%+ SLA）

**技术架构**
- 后端：Go/Java（推荐 Go）
- 数据库：PostgreSQL（元数据）+ Redis（缓存）
- HSM：PKCS#11 兼容设备
- API：RESTful + gRPC

**实现难度**: ⭐⭐⭐⭐（高）

**团队需求**: 5-8人
- 密码学专家：1人
- 后端工程师：3-4人
- 安全工程师：1-2人
- DevOps：1人

**开发周期**
- MVP：3-4个月
- 生产化：6-8个月
- 社区推广：9-12个月

**社区价值**: 高（为所有上层产品与生态提供可信密钥基础设施）

**可持续支持**
- 社区捐赠与公共资助（如以太坊基金会、Web3 基金）
- 企业/机构赞助节点与审计资源
- 文档、教程与翻译贡献，降低集成门槛

**可持续支持**
- 捐赠或赞助 HSM 设备与实验室环境

### 阶段二：核心技术层（12-18个月）

#### 3. MPC（多方安全计算）

**产品定位**
- 核心技术，赋能多签钱包和托管方案
- 去中心化密钥管理的技术基础

**目标客户**
- 需要去中心化密钥管理的机构
- 对安全性要求极高的企业
- 多签钱包服务提供商

**核心功能**
- 阈值签名（TSS - Threshold Signature Scheme）
- 密钥分片生成与管理
- 分布式签名协议
- 密钥轮换（无需重新分发）
- 性能优化（低延迟、高吞吐）

**技术架构**
- 协议：GG18、GG20、FROST
- 通信：gRPC、WebSocket
- 存储：分布式密钥分片存储
- 网络：P2P 或中心化协调

**实现难度**: ⭐⭐⭐⭐⭐（极高）

**团队需求**: 8-12人
- 密码学博士：2-3人
- 算法工程师：3-4人
- 分布式系统工程师：3-5人

**开发周期**
- Phase 1（2/3 阈值签名）：6个月
- Phase 2（通用阈值签名）：6个月
- Phase 3（性能优化与生产化）：6个月
- **总计**: 18个月

**技术路线**
1. **Phase 1**: 实现 2-of-3 阈值签名
   - 支持 ECDSA、EdDSA
   - 基础性能：签名延迟 < 500ms
   - 支持 BTC、ETH 等主流链

2. **Phase 2**: 通用阈值签名
   - 支持任意 M-of-N
   - 支持更多签名算法
   - 支持更多区块链

3. **Phase 3**: 生产化优化
   - 性能优化：签名延迟 < 200ms
   - 安全加固：侧信道攻击防护
   - 高可用：容错和恢复机制

**社区价值**: 极高（形成差异化的去中心化安全能力，惠及所有钱包与应用）

**可持续支持**
- 研究资助（学术合作、基金会拨款）
- 开源贡献者计划与长期维护人激励
- 安全审计、测试网基础设施由社区协作承担

**竞争优势**
- 无需私钥重构（vs 传统多签）
- 更好的用户体验
- 更高的安全性

### 阶段三：钱包产品线（18-30个月）

#### 4. 自托管钱包

**产品定位**
- 入门级产品，建立用户基础
- 用户完全控制私钥

**目标客户**
- 个人用户
- DeFi 用户
- 加密货币爱好者

**核心功能**
- 私钥生成与存储（本地加密）
- 多链支持（BTC、ETH、EVM 链、Cosmos 等）
- 助记词备份与恢复
- 交易签名与广播
- DApp 连接（WalletConnect）
- NFT 管理

**技术架构**
- 前端：React/React Native
- 后端：无（或轻量级服务）
- 区块链集成：多链 SDK
- 存储：本地加密存储

**实现难度**: ⭐⭐⭐（中等）

**团队需求**: 4-6人
- 前端工程师：2人
- 移动端工程师：1-2人
- 区块链集成：1-2人

**开发周期**
- MVP：3-4个月
- 完整版：6-8个月

**社区价值**: 中（教育更多用户安全自托管，并为上层钱包提供参考实现）

**可持续支持**
- 社区翻译、文档、UX 贡献
- 去中心化基金或社区 Treasury 覆盖基础设施费用

#### 5. 多签钱包

**产品定位**
- 企业级钱包，基于 MPC 或传统多签
- 团队和机构资产管理

**目标客户**
- 企业
- DAO（去中心化自治组织）
- 团队管理的加密资产账户
- 基金和投资机构

**核心功能**
- 多签名协议（MPC 或传统多签）
- 权限管理（M-of-N 配置）
- 审批流程（交易审批、地址白名单）
- 多链支持
- 角色管理（管理员、审批者、观察者）
- 审计日志

**技术路线**
- **方案A**: 传统多签（较快，3个月）
  - 优点：实现简单，兼容性好
  - 缺点：需要私钥重构，用户体验差

- **方案B**: MPC 多签（较慢，6-9个月，但更优）
  - 优点：无需私钥重构，用户体验好，安全性高
  - 缺点：实现复杂，需要 MPC 技术支撑

**推荐**: 优先实现 MPC 多签（与 MPC 产品协同）

**技术架构**
- 后端：基于 MPC 服务
- 前端：Web + Mobile
- 审批系统：工作流引擎

**实现难度**: ⭐⭐⭐⭐（高）

**团队需求**: 5-7人
- 后端工程师：2-3人
- 前端工程师：2人
- 产品经理：1人
- 测试工程师：1人

**开发周期**
- 传统多签：3-4个月
- MPC 多签：6-9个月

**社区价值**: 高（为 DAO、团队、自组织社区提供透明的资产治理能力）

**可持续支持**
- 社区治理金库出资维护
- 合作伙伴投入审计、合规、运营支撑

## 实施路线图

### Phase A · 基础设施与核心协议
- 完成 `go-kms` / `go-mpc` MVP，并开放测试网。
- 建立贡献者指南、安全披露制度与公共演示环境。
- 发布自托管钱包示例，验证端到端体验。

### Phase B · 钱包矩阵与治理工具
- 推出基于 MPC 的多签钱包与智能钱包，完成安全审计。
- 托管钱包进入社区灰度，建设自动化监控与报警。
- 引入治理/提案工具，确保资金与配置透明。

### Phase C · 硬件与生态拓展
- 打磨托管钱包正式版，与 KMS/MPC 深度集成。
- 众筹/共建开源硬件钱包，并完成必要认证。
- 建立国际化志愿者节点与合作伙伴网络。

### 社区协作节奏
- **季度 OKR**：每季度初开放规划 Issue + 社区会议，总结阶段目标。
- **双周例会**：同步核心模块进展，邀请新贡献者分享。
- **发布周期**：KMS/MPC 采用 6 周节奏，钱包/前端采用双周节奏。
- **测试活动**：每阶段至少组织一次公开测试/攻防演练。
- **治理流程**：采用 KIP（KashGuard Improvement Proposal）制度，保证路线更新可追溯。

- ⭐⭐: 中（行业标准认证）
- ⭐⭐⭐: 高（金融级合规）
- ⭐⭐⭐⭐: 极高（硬件安全认证）
- ⭐⭐⭐⭐⭐: 极高（金融牌照 + 多重认证）

**市场难度评估标准**:
- ⭐⭐: 低（企业刚需，销售周期短）
- ⭐⭐⭐: 中（有需求，需要市场教育）
- ⭐⭐⭐⭐: 高（竞争激烈，需要差异化）

## 社区与资源规划

### 核心角色与工作组

- **核心维护者（Core Maintainers）**：负责架构演进、版本发布、KIP 审核。
- **协议与密码学组**：聚焦 MPC/TSS 研究、实现与安全证明。
- **基础设施与 DevOps 组**：维护测试网、CI/CD、监控与发布。
- **钱包体验组**：覆盖自托管、多签、智能钱包等前后端实现。
- **硬件与安全组**：推进开源硬件、固件、安全认证与攻防演练。
- **文档与本地化组**：维护多语言文档、教程、社区活动与内容。

### 贡献路径

1. **Onboarding**：通过「Good First Issue」「文档 Errata」快速入门。
2. **工作组成员**：参与双周例会、认领中型任务、协助需求评审。
3. **模块 Owner**：负责某个子模块的 Roadmap、Review 与发布。
4. **社区顾问**：来自高校/机构的专家，提供技术或合规顾问支持。

### 治理与决策

- 采用多签委员会 + KIP 提案流程，重大变更需社区投票。
- 每半年进行一次 Maintainer 轮换/增补，确保去中心化治理。
- 所有资金流向（捐赠、赞助、赏金）公开透明，接受社区监督。

## 社区生态策略

### 目标社群与协作方式

#### 1. 机构与节点运营者（B2B 场景）

- **需求**: 合规、安全、可审计的密钥与签名能力。
- **协作方式**:
  - 共同制定开源合规模板、审计清单。
  - 以赞助节点、硬件、审计等形式反哺社区。
  - 分享实践案例，沉淀参考架构与最佳实践。

#### 2. 自托管与终端用户社群（B2C 场景）

- **需求**: 易用、可恢复、可信的个人钱包体验。
- **协作方式**:
  - 参与测试网、翻译、文档、教学内容。
  - 组织本地化 meetup / workshop，推广安全自托管。
  - 贡献 UI/UX、无障碍、可用性改进。

#### 3. 开发者与集成伙伴（B2D 场景）

- **需求**: 稳定的 API、SDK、脚手架、示例项目。
- **协作方式**:
  - 举办 Hackathon、Builder Camp、技术分享。
  - 维护开放的 RFC/KIP，鼓励跨项目协作。
  - 建立集成伙伴目录，推动互审与互认证。

### 推广与教育节奏

- **内容**: 白皮书、技术笔记、案例研究、播客/直播。
- **活动**: 每季度社区大会、区域工作坊、线上读书会。
- **合作**: 与高校、研究机构、基金会共建实验室/资助项目。
- **透明度**: 定期发布治理报告、路线图回顾、安全公告。

### 竞合与差异化

- **与商业 KMS/托管的区别**: 完全开源、可自部署、MPC 原生、社区治理。
- **与传统钱包的区别**: 阈值签名、治理/审计能力、端到端可验证。
- **与其他开源项目的合作**: 共享协议实现、共建标准、互相复用组件。
